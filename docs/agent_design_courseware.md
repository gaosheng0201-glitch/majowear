## 教学模块一：前置分词、意图分类与分级智能路由 (Routing Design)

### 1.1 为什么要进行意图分类与分词？
在包含工具链（Function Calling）与联网搜索（Search Grounding）的复杂 Agent 系统中，如果直接将所有工具和搜索能力全量堆叠给主模型，会引发两个致命问题：
1.  **API 冲突与混乱**：模型在需要回答事实问题时误触发了生图工具，或者在生图时误触发了搜索，造成极高额的无效 Token 与算力浪费。
2.  **上下文污染**：搜索引擎返回的网页摘要会干扰工具调用的参数组装，降低参数准确率。

### 1.2 分级智能路由设计
系统在 [route.ts](file:///d:/majowear/src/app/api/agent/generate/route.ts#L492-L556) 中实现了一个极速的 **“意图分流器 (Intent Router)”**，使用 `gemini-3.5-flash` 将用户 Prompt 快速分流为三种动作状态：
*   **DEEP_THINK（深度推理）**：用户在询问工艺原因、设计对比或工艺推导。路由至 **`gemini-3.1-pro-preview`**，开启原生 Thinking 模式（思维链），获取极高的逻辑深度。
*   **TOOL（工具执行）**：用户明确要求设计衣服或保存面料/风格。路由至直连工具链的 `gemini-3.5-flash`，以降低响应时延。
*   **SEARCH（知识检索）**：日常闲聊或问答。路由至开启了 **Google Search Grounding** 的 Flash 模型，返回带有真实引用来源的网页摘要。

## 教学模块二：NLP 语义比对与分歧决策反馈环路 (Conflict Loop)

当设计师激活了侧边栏的“A 面料”（如奢华羊绒），却在聊天栏里对 Agent 说：“帮我设计一版专业潜水服”时，系统应当如何表现？

### 2.1 传统设计的弊端
传统的 AI 助手通常会装聋作哑，顺着用户的提示词强行用羊绒材料生图，结果生出了一款“毛绒绒的潜水衣”。这破坏了服装设计的“顺向物理约束一致性”。

### 2.2 决策拦截器提示词与分词匹配算法
系统在后台构建了 `detectAndResolveConflict` 模块：
1.  **上下文比对**：将当前侧边栏**已激活的面料/风格属性**、**项目中所有可用资产的元数据列表**以及**用户当前 Prompt** 一起塞给拦截器。
2.  **分词与 NLP 相似度决策**：
    *   **静默匹配**：若大模型判定用户提到的面料（如 "羊毛"）与项目里某张已存在的卡片（如 "Merino"）语义相似度高于 95%，则**静默替换** ID，不弹窗打扰用户。
    *   **分歧抛出**：若输入与当前激活属性存在显式矛盾（如羊绒 vs 潜水衣），则拦截生成流，动态生成多选题卡片。
3.  **防重复重置逻辑**：当设计师做出选择后，前端重新触发提交并携带 `conflictResolved = true` 标头，告知后端直接放行，避免陷入“决策循环”。

## 教学模块三：助手 Agent（Sub-agent）前置抢跑与资产遗传机制

当设计师在决策卡片中选择了“使用专业潜水料 (Neoprene)”，但项目中原本没有这张面料卡时，参数值以非 UUID 的 `custom_neoprene` 形式上传。

### 3.1 什么是子 Agent 抢跑机制？
为了保持数据一致性，设计图绝不能使用一个悬空的概念名称（`custom_neoprene`）进行生成。
1.  **抢跑启动**：后端在主设计任务（生图）启动前，识别到 ID 并非 UUID。
2.  **子 Agent 激活**：唤醒 **“助手 Agent”**（`gemini-3.5-flash`），根据面料的字面语义和设计主题，逆向推导出该面料的物理特性规格（如成分 `80% Neoprene`、克重 `320gsm`、硬挺悬垂度、双面针织肌理等）并生成 Optimized Rendering Prompt。
3.  **Fail-fast 写入**：将新推导的属性卡写入项目数据库，换取正式的 UUID。
4.  **外键绑定**：主 “设计 Agent” 继任，使用这个正式 UUID 和物理属性去设计款式并进行关联，确保生成的款式卡在数据库中完美归档，侧边栏能够自动刷出新卡。

### 3.2 历史遗传上下文 (Lineage & Inherit)
当设计师在聊天中说：“把 `@简约宽松卫衣` 换成皮革面料”时：
*   **Mention 标签化 (Pill 标签)**：前端把 `@简约宽松卫衣` 转换为不可分割的富文本胶囊标签，防止其因用户退格而碎裂。
*   **属性遗传**：后端从数据库拉取该卫衣的结构化参数（版型、领口、closures、details），将除面料外的所有款式设计参数作为“遗传因子”传递给 Gemini。
*   **图像遗传**：提取该卫衣之前的图像，作为 Gemini `image_to_image` 或 `editImage` 的参考底图输入，确保款式细节的演进是递进的，而不是每次都随机乱画。

## 教学模块四：流式响应协议与双 Agent 工作透明度 (UX Design)

在生成设计款式期间，涉及到“助手 Agent 写库”和“设计 Agent 调用生图”（总共耗时约 6~8 秒）。如何避免设计师在黑盒等待中失去耐心？

### 4.1 自定义 JSON Lines 流式传输协议
系统抛弃了传统的“纯文本流”，构建了结构化 JSON 行流式响应协议。后端在不同的工作流阶段推送包含 `type` 字段的事件：
*   `{ "type": "status", "status": "waiting_subagent_fabric" }`：主 Agent 进入等待。
*   `{ "type": "status", "status": "subagent_generating_fabric" }`：助手 Agent 启动工作。
*   `{ "type": "created_fabric", "data": { ... } }`：**【高光点】子资产提前下发包**。
*   `{ "type": "status", "status": "rendering", "target": "garment" }`：主 Agent 启动生图。

### 4.2 协作透明度与骨架屏幻化 UX 交互
前端在 `AgentChat.tsx` 侦听到状态流时，提供精美的多 Agent 视觉反馈：
1.  **角色展示**：主 Agent 明确指代为 **设计 Agent**，子 Agent 明确指代为 **助手 Agent**。
2.  **发光协作条 (SVG)**：当助手 Agent 接入时，前端使用 SVG 精致旋转动画（`<Sparkles className="animate-spin text-primary" />`）展示 `助手 Agent 协同中` 的专属面板。
3.  **骨架屏到实卡平滑过渡**：
    *   在推导面料参数时，协作面板内渲染与资产库样式 100% 吻合的原生**面料卡骨架屏**。
    *   当 stream 解析器读取到 `created_fabric` 数据块时，Zustand store 立即刷入该面料并激活，同时**协作面板内的骨架屏在原地以微秒级的过渡动效瞬间化为真实的交互式面料卡**。
    *   此时设计 Agent 的状态更新为 `正在设计款式与渲染效果图...`，继续拉起款式骨架屏。这让用户能够全程、清晰、富有掌控感地看到资产是如何被创造和利用的。

## 教学模块五：教学大纲建议与复现动手步骤

本节可直接用于教师编写教案与实验指导书：

### 课时安排与动手实验 (4 课时)

*   **第 1 课时：Agent 的意图分流与分词基础**
    *   *理论讲解*：工具调用的机制、意图选择算法、幻觉与防冲突设计。
    *   *动手练习*：在 Postman 中调用 `/api/agent/generate` 并输入冲突文本，观察 Gemini NLP 分析返回的 JSON。
*   **第 2 课时：双 Agent 抢跑生成与 Fail-fast 表级关联设计**
    *   *理论讲解*：子 Agent 在分布式/微服务协同中的设计逻辑、Fail-fast 原则与静默降级的权衡。
    *   *动手练习*：编写 `generateFabricCardSpecs` 并调用 Supabase 将推导参数入库，检查是否能无报错返回正式 UUID。
*   **第 3 课时：工作流流式事件协议与 ReadableStream 实现**
    *   *理论讲解*：流式传输在重载 AI 模型响应中的人机工程学应用。
    *   *动手练习*：使用 `ReadableStream` 改造传统的 Next.js API，让其能通过 HTTP 持续输出具有不同 `type` 的 JSON 数据帧。
*   **第 4 课时：前端 React 协同看板与状态变幻交互**
    *   *理论讲解*：骨架占位屏（Skeleton Screen）的设计原则、协作工作区透明度设计。
    *   *动手练习*：在 React 中实现解析器接收 `created_fabric` 等数据，让局部的骨架屏随着状态的变更平滑淡出，替换为可交互的面料卡。
